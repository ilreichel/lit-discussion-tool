  Purpose: Develop an interface complying with the project speifications.

  Prompt text:

  Project Specification
1. Overview
Build an educational discussion platform for humanities-based literature and scientific textbooks. The platform is intended for secondary and undergraduate education and supports classroom-based reading analysis using quotations and themes/tags.
The application should provide:
a classroom landing page,
a discussion post page organized by book and chapter,
a data visualization page for theme/tag relationships,
a theme page showing posts associated with a selected theme/tag.
The implementation will be completed in opencode

2. Core Purpose
The system must allow students to:
join a preexisting class,
view assigned books,
read discussion posts,
create discussion entries tied to chapters,
associate quotations with themes/tags,
explore theme/tag relationships through visualization.
The system must support multiple classes in the future, even though the demo experience only requires one preexisting class.

3. Demo Scope and Access Model
3.1 Login
The login system will be demonstrated using a default account
No password, authentication provider, or recovery system is required for the demo.
The demo must include exactly one student account experience.
The temporary default user account should be named “student1” and entries created will use this
There should not be an educator-only view in the demo.
Educator features are not implemented for the demo and may be ignored for now.
3.2 Preexisting Demo Data
The demo must include:
a preexisting class named “Intro to English Lit”
a preloaded book assigned to that class: Brave New World by Aldous Huxley
The student should enter directly into the experience with this class and book already available.

4. Information Architecture
The application must contain four primary pages:
Home
Discussion
Visualizations
Themes
Each page must follow the layout and behavior requirements below.

5. Data Model Requirements
The implementation must use clearly defined, strongly declared, and documented data structures. All classes, method signatures, DTOs, and collection types must be explicitly declared and documented.
5.1 Core Entities
The system should define data models for at least the following entities:
User
Student
Class
Book/Textbook
Chapter
DiscussionPost
Quote
Theme
ThemeFrequency
VisualizationNode
VisualizationEdge
5.2 Discussion Post Fields
Each discussion post must store:
student identity
associated class
associated book
chapter number or chapter identifier
publication time / creation timestamp
quote(s) used
manually entered tag(s)/theme(s)
general discussion text
any system-generated metadata needed for analytics
5.3 Quote and Tag Rules
Quotes must always be autodetected by the system.
Tags/themes are manually entered by the user.
The words themes and tags are used interchangeably throughout this specification.
A discussion post must contain at least one quotation before submission.
Each quotation must be greater than two words.
Each quotation must have a minimum of one tag.
Multiple tags per quote are allowed.
If a quote already has one or more tags, the user does not need to enter another tag, but may still add additional tags.
Repeat quotations are allowed.
When a repeat quotation is used, the ThemeFrequency of the associated tags must increase by one each time the quotation is reused.
A tag created by another user must not be removable.
If a quotation exists without an associated tag, the system must require a tag to be assigned before the post can be submitted.
5.4 Naming and Modularity Standards
Use descriptive, consistent names for variables, methods, classes, and UI elements.
Avoid abbreviations unless they are universally clear.
Separate concerns into distinct modules for:
login/session handling,
class management,
discussion post management,
quote detection and storage,
tag/theme analytics,
visualization generation,
page routing/navigation.
Keep business logic out of presentation components whenever possible.
Use reusable helper functions for:
quote detection,
tag aggregation,
sorting posts by chapter and publication time,
building visualization nodes and edges.

6. Page Specifications
6.1 Home Page
The Home page functions as the landing dashboard.
Required Layout
Top-Left:
Large text saying “Discussion Tool” as the title of the website
Top-right corner:
username login input
login action control
Below the login area:
section title: My Classes
Below “My Classes”:
a visible demo class card or box
the demo class must be preloaded and accessible
Behavior
Selecting the demo class should navigate the user into the class experience.
The page should clearly communicate class enrollment status.
The layout must remain usable on smaller screens.

6.2 Discussion Page
The Discussion page must present book-based discussion content organized by chapter and then by publication time.
Required Layout
Left sidebar:
list of book titles
nested or related chapter list for each book
Main content area:
selected book title at the top
discussion posts sorted first by chapter number, then by publication time
Top-right corner:
a rectangular button labeled Data
Bottom-right corner:
a rectangular Create Entry button
Behavior
Clicking a book should jump to the start of that book section.
Clicking a chapter should jump to the start of that chapter section.
Clicking the Data button should navigate to the data visualization page.
Clicking Create Entry should open the entry creation pop-up.
Discussion posts must appear in chapter order, then by publication time within each chapter.
Each post should visibly include:
student name or identifier,
discussion text,
automatically detected quote(s) within discussion,
manually entered tag(s)/theme(s) required pop-up prompt for quotes,
publication time.
Submission Rules
A post cannot be submitted unless it contains at least one quotation.
Each quotation must contain more than two words.
Each quotation must have at least one tag.
If a quotation already has associated tags, additional tags are optional.
Repeat quotations must increase the ThemeFrequency count for associated tags.
Tags created by another user cannot be removed.
Behavior Safeguards
If a chapter has no posts, display a clear empty-state message.
If a book has no chapters loaded, show a structured placeholder instead of a blank area.
If a post is missing required fields, prevent submission and explain what is missing.
Preserve user input if validation fails.

6.3 Visualizations Page
The Visualizations page must display an interactive semantic graph of themes/tags.
Required Layout
Bottom-right corner:
a checkbox panel containing:
“Select All”
all book titles
Top-left corner:
back button
Main area:
interactive graph visualization similar to the provided reference image
Visualization Behavior
The graph must:
represent themes/tags as nodes,
use semantic clustering to group related themes,
use Cosine Similarity as the Semantic Similarity Score,
use a Force-Directed Graph algorithm,
visually distinguish related themes using the same color from the palette to represent a cluster,
scale node size according to recurrence/frequency,
support zooming and panning,
hide or de-emphasize minor nodes when zoomed out to reduce clutter,
prevent high-frequency tags from covering the entire screen,
allow node selection,
allow clicking a node to navigate to the related theme page or filtered theme view.
Scope Control
The checkbox panel must allow filtering by:
all books, or
one or more selected books.
The graph must refresh dynamically based on the selected scope.
The selected scope should also affect node frequency counts if the visualization is scoped to selected books only.
Visualization Performance Requirements
Use incremental rendering or efficient graph updates for large datasets.
Avoid re-rendering the entire graph when only one filter changes.
Cap visible node density or apply progressive simplification if the graph becomes too dense.
Lazy-load graph data if necessary.
Visualization Fail-Safes
If semantic clustering cannot complete, fall back to a simpler grouping strategy based on exact tag matching.
If graph data is missing or invalid, display a stable fallback state rather than crashing.
Prevent infinite layout loops or unbounded force simulations.
Protect against overlapping labels by truncating, hiding, or collapsing labels at low zoom levels.

6.4 Themes Page
The Themes page must show theme-specific discussion content.
Required Layout
Bottom-right corner:
checkbox panel with:
“Select All”
all book titles
Top-left corner:
back button
Main content area:
theme name at the top
discussion posts listed below, sorted by chapter and then publication time
Behavior
Selecting a theme should show all posts associated with that theme.
Posts should be grouped or sorted by chapter, then by publication time.
Filtering by book must update the displayed posts.
Empty-state handling must be provided when a theme has no posts.

7. Data Visualization Logic
The visualization should use semantic associations to connect themes/tags. The implementation should interpret semantic mapping as follows:
Themes/tags with similar meaning or frequent co-occurrence should be placed near each other.
Semantic relation should be shown through proximity to other nodes
Node should be in the form of circles with readable labels that adjust with sizing
Theme clusters should be visually grouped.
Repeated themes should appear larger or more prominent.
More frequent themes should remain readable at lower zoom levels.
Nodes should be clickable and should reveal theme-related discussion posts.

8. Design Requirements
8.1 Color Palette
Use the following palette consistently across the application:
#335c67
#fff3b0
#e09f3e
#9e2a2b
8.2 Visual Style
The design must be intuitive, clean, and visually appealing.
Use strong contrast for readability.
Maintain consistent spacing, typography, and card styling.
Reserve accent colors for interactive elements, labels, and highlights.
Keep the layout readable on both desktop and smaller screens.
Assign the same palette color to similar themes within the same cluster.
8.3 Layout Consistency
Use a shared layout system across pages so navigation, buttons, and side panels behave consistently.
Keep primary navigation predictable.
Keep interactive controls in the same general locations across pages whenever possible.

9. Technical Requirements
9.1 Stack
Backend: Java
Frontend: HTML
Database: PostgreSQLDynamic
9.2 Architecture Clarification
The implementation should be structured so that the frontend and backend remain modular and maintainable. The system should support multiple classes in the future, even though the demo uses only one class.
9.3 Code Quality Standards
Use strong typing throughout the backend.
Document all public classes, methods, and data models.
Use modular architecture.
Keep functions single-purpose.
Prefer small, testable service classes over large monolithic files.
Add validation at every user input boundary.
Use consistent error handling conventions.

10. Error Handling and Fail-Safes
The implementation should anticipate and safely handle the following problems:
Data and Input Problems
missing username
invalid chapter selection
empty discussion post submission
missing quote text
quote shorter than or equal to two words
missing tag/theme
duplicate quotes
duplicate book titles
invalid or null class references
UI and Rendering Problems
graph layout overload
too many nodes or edges for the viewport
empty books list
no discussion posts for a selected filter
failed navigation between pages
System Problems
unavailable or partially loaded visualization data
inconsistent counts between book-level and class-level tag statistics
failed save operations
partial page data loads
Required Safeguards
validate inputs before persistence
show user-friendly error messages
preserve previously entered user input when validation fails
never crash the entire page due to one bad record
use fallback UI states for missing or incomplete data
log recoverable errors for debugging without exposing internal stack traces to users

11. Performance Guidance
To improve efficiency and reliability:
Precompute recurring tag counts where possible.
Cache class-wide and book-specific analytics.
Avoid recomputing graph layout on every small interaction.
Paginate or virtualize long discussion lists if necessary.
Defer heavy visualization processing until the page is visible.
Load only the currently selected book or theme content when feasible.
Use efficient data structures for lookup by book, chapter, quote, and tag/theme.

12. Accessibility and Usability Recommendations
Although not originally specified, the following should be included to improve quality:
keyboard-accessible controls
visible focus states
sufficient color contrast
text alternatives for icons
clear button labels
predictable navigation
readable font sizes
responsive layout behavior

13. Clarifications Before Development
Normalize quotes for matching
Multiple tags per quote should be displayed as separate chips
Chapter jumps should be instant
